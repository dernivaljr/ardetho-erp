# Estrutura oficial do projeto Ardetho ERP

## 1. Visão geral da arquitetura

Como vocês vão usar apenas **HTML, CSS e JavaScript**, a melhor abordagem é montar o projeto com:

* páginas separadas por contexto;
* arquivos CSS divididos por responsabilidade;
* JavaScript modular por área;
* dados simulados centralizados;
* componentes visuais reutilizáveis.

Mesmo sem framework, dá para trabalhar de forma bem profissional.

---

# 2. Estrutura de páginas

Eu sugiro separar em **área pública** e **área interna**.

# 4. Função de cada arquivo CSS

Para o projeto não repetir o problema comum de “um CSS gigante e confuso”, cada arquivo precisa ter função clara.

## `variables.css`

Vai guardar:

* cores;
* fontes;
* espaçamentos;
* bordas;
* sombras;
* tamanhos padrão.

Exemplo:

* `--color-bg-main`
* `--color-primary`
* `--radius-md`
* `--shadow-soft`

---

## `reset.css`

Vai normalizar:

* margens;
* paddings;
* box-sizing;
* listas;
* links;
* botões;
* inputs.

---

## `global.css`

Vai conter estilos globais:

* body;
* tipografia;
* títulos;
* textos;
* containers;
* utilitários simples.

---

## `layout.css`

Vai controlar:

* sidebar;
* topbar;
* grid principal;
* estrutura das páginas;
* largura de conteúdo;
* alinhamento geral.

---

## `components.css`

Vai concentrar os componentes reutilizáveis:

* botões;
* cards;
* badges;
* inputs;
* tabelas;
* modais;
* dropdowns;
* toasts;
* filtros;
* paginação.

Esse arquivo vai ser um dos mais importantes.

---

## `public.css`

Vai estilizar a área pública:

* hero section;
* seções institucionais;
* CTA;
* planos/módulos;
* contato;
* navbar pública;
* footer.

---

## `auth.css`

Vai cuidar da tela de login:

* box de autenticação;
* campos;
* botão;
* mensagens de erro;
* layout dividido, se houver.

---

## `dashboard.css`

Vai cuidar do dashboard:

* cards de resumo;
* widgets;
* área de atividade recente;
* gráficos simulados;
* resumo financeiro.

---

## `pages.css`

Vai conter estilos específicos das páginas internas:

* clientes;
* produtos;
* vendas;
* financeiro;
* relatórios;
* módulos;
* configurações;
* perfil.

---

## `responsive.css`

Vai conter as media queries do sistema todo:

* tablet;
* mobile;
* ajustes da sidebar;
* tabelas responsivas;
* cards empilhando.

---

# 5. Função de cada arquivo JavaScript

Agora a estrutura lógica.

## `main.js`

Arquivo geral do sistema:

* inicializações globais;
* tema;
* comportamento genérico;
* chamadas de funções comuns.

---

## `auth.js`

Vai controlar:

* login fake;
* logout;
* validação de usuário;
* usuário logado no `localStorage`;
* redirecionamento simples.

---

## `navigation.js`

Vai controlar:

* destaque da página ativa;
* toggle da sidebar;
* dropdown do usuário;
* comportamento do menu.

---

## `dashboard.js`

Vai renderizar:

* cards do dashboard;
* resumo de métricas;
* atividades recentes;
* dados simulados;
* widgets.

---

## `clients.js`

Vai controlar:

* listagem de clientes;
* busca;
* filtros;
* modal de novo cliente;
* edição/exclusão simulada.

---

## `products.js`

Vai controlar:

* produtos e serviços;
* categorias;
* status;
* busca e filtros;
* tabela ou cards.

---

## `sales.js`

Vai controlar:

* pedidos;
* status;
* datas;
* filtros;
* resumo comercial.

---

## `financial.js`

Vai controlar:

* contas a pagar;
* contas a receber;
* vencimentos;
* saldos simulados;
* resumos financeiros.

---

## `reports.js`

Vai cuidar de:

* relatórios simulados;
* cards analíticos;
* filtros de período;
* exportação fictícia.

---

## `modules.js`

Vai cuidar da página mais estratégica do projeto:

* módulos ativos/inativos;
* toggle visual;
* descrição de cada módulo;
* persistência no `localStorage`.

---

## `settings.js`

Vai controlar:

* preferências do sistema;
* tema;
* configurações visuais;
* preferências de exibição.

---

## `profile.js`

Vai renderizar:

* dados do usuário;
* cargo;
* e-mail;
* último acesso;
* edição simulada.

---

## `data.js`

Esse será o “mini banco” do projeto.

Vai conter:

* usuários fake;
* clientes fake;
* produtos fake;
* pedidos fake;
* dados financeiros fake;
* relatórios fake;
* módulos disponíveis.

Esse arquivo é essencial.

---

## `storage.js`

Vai abstrair o uso do `localStorage`:

* salvar;
* buscar;
* remover;
* resetar dados.

Isso deixa o código mais organizado do que usar `localStorage` solto em vários arquivos.

---

## `utils.js`

Vai ter funções auxiliares:

* formatar moeda;
* formatar data;
* gerar ID simples;
* filtrar listas;
* validar campos;
* exibir notificações.

# 9. Estratégia de navegação

Como ainda não existe back-end, recomendo duas regras simples:

## Regra 1

Toda página interna verifica se existe usuário logado no `localStorage`.

Se não existir:

* redireciona para `login.html`.

## Regra 2

O nome da página atual deve ficar destacado na sidebar.

Isso deixa a navegação muito mais profissional.

# 11. Prioridade visual

Se vocês quiserem replicar o nível do projeto X-Men, a prioridade não é “fazer tudo logo”.

A prioridade é:

1. consistência visual
2. layout forte
3. navegação limpa
4. componentes bem feitos
5. detalhes de UX

Um sistema com menos páginas, mas bem construído, parece muito melhor do que um sistema enorme e inconsistente.
